The Self-Improving Agent Loop: How amux Gets Smarter Every Run

One command. Parallel agents. A verify gate. A constraints note that accumulates lessons. Run 50 is smarter than run 1 — automatically.

The Problem with Agent Swarms

Running agents in parallel is solved. The hard part is making them learn from each run without retraining a model. Most setups do the same thing on run 50 as they did on run 1. The context window closes, the lessons disappear, and you're back to square one.

amux solves this with a three-part structure: a mission note, a state note overwritten every tick, and a constraints note that only ever grows. The constraints note is the memory. Every run appends what it learned. The next run loads it before doing anything else.

One Command to Start

/orchestrate -g "validate batch objects in MVS, run hybrid search evals across RRF and weighted variants, measure precision/recall/latency, fix issues as found" -s "ts-gke, backend"

That single command:

  • Derives a slug from the goal (mmv2-retrieval-eval)
  • Fills the v2 loop template with your goal, the authorized session set, and inferred issue prefixes (TG-, BACKE-)
  • Creates three notes: mission brief, state (overwritten each tick), constraints (append-only)
  • Wires a scheduler entry that fires every 2h at the orchestrator session

From that point the loop runs without human input. Each tick: load state → peek sessions → think → act → verify to prod → distill a lesson → persist.

The Closed Loop

Most orchestration systems are one-shot: you fire a prompt, agents do work, you check the output. amux's loop is different. The orchestrator session runs on a schedule, loads its own state from a note, observes what changed, decides what to do next, and writes its conclusions back before it stops.

Two thinking gates enforce deliberate reasoning:

  • Before acting: diagnose what's actually blocked vs. just slow, check for cross-session conflicts, decide whether doing nothing is better than acting
  • Before persisting: distill what happened into 0–2 constraint lines — only write ones that would have changed behavior this tick

The constraints note is append-only. It starts with one seeded rule and grows from there:

2026-06-17 — stage explicit git paths only, never git add -A.
Reason: commit 5407ac1473 swept another session's staged deletions
into the wrong commit and broke import mvs.agent repo-wide (AMUX-1315).

2026-06-19 — verify each batch object exists in MVS before running
evals. Silent absence looks like a precision problem but is actually
a data-pipeline problem. Check existence first, eval second.

By run 10, the orchestrator is carrying a decade of hard-won rules into every tick automatically.

Access Control Is a First-Class Feature

Every loop defines exactly which sessions the orchestrator can interact with — peek, send, or post board issues. Any session outside that table gets a board item for the human instead of a direct message.

This matters in practice. During a real incident, our orchestrator couldn't deliver a standing rule to a session outside its authorized set. That limitation turned out to be the strongest argument for the durable fix: a behavioral rule you can't deliver to every session is a rule that doesn't work. The loop escalated to the board; the human made the policy call.

Issues vs. Tasks

The amux board serves two distinct roles inside a loop. Conflating them is how you lose track of what's real work and what's just coordination noise.

  • Prefixed issues (TG-, BACKE-, AMUX-): durable, survive context compaction, multiple sessions can reference the same one, closed only when fully verified in prod
  • Tick-level tasks: ephemeral routing within a single tick — "confirm deploy SHA for TG-83" — reference issues but aren't issues themselves

The verified standard for closing an issue is strict: committed on main, CI green, staging E2E green, prod deployed, proven with real traffic evidence, and the amux issue closed. Not "deploy succeeded." Real evidence.

The Skills Tab

The /orchestrate command is one of dozens of slash skills available in amux. The new Skills tab gives you a searchable browser for all of them — custom skills stored in SQLite, project-level .claude/commands/ files, and built-in Claude Code commands.

Every skill card is expandable. Click ▾ to see the full markdown body: headings, bullets, and code blocks each with their own Copy button. One click to grab any example command and paste it into a session.

Expand All Active Sessions

Small feature, high leverage: the ⚡ button in the session list expands every running non-idle session at once. Same button in Workspace mode adds terminal panes for all active sessions simultaneously. When you have 7+ agents in flight, "show me what's happening" should be one click.

The Primitives Were Already There

This pattern doesn't require a specialized multi-agent platform. Here's how the pieces map:

Inter-agent messagingPOST /api/sessions/:name/send
Output observationGET /api/sessions/:name/peek
Skill / constraint storageNotes API (/api/notes/:slug)
Background loopScheduler (/api/schedules)

The scheduler is the last primitive. It was already built. /orchestrate just wires everything together.

What's Next

/orchestrate list to see all active loops and their last-tick state. /orchestrate stop <slug> to disable a scheduler entry and archive its notes. Right now you manage those manually via the Scheduler tab — automating it is the obvious next step.

We're also watching whether the constraints note format holds up over 20+ runs. The hypothesis is that it becomes genuine organizational memory that survives session restarts and context compaction. Early evidence from our production orchestrator suggests yes.

Get started with amux

Run a self-improving agent loop in your own codebase. Open source, Python 3 + tmux.

git clone https://github.com/mixpeek/amux && cd amux && ./install.sh
amux serve  # → https://localhost:8822
# Then: /orchestrate -g "your goal" -s "session-a, session-b"
